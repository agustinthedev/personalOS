Crear una nueva mini app llamada Portfolio dentro de Personal OS.

Contexto del proyecto
=====================

Personal OS es una app Next.js con App Router, Prisma + SQLite, TypeScript y Tailwind. El proyecto está organizado como un conjunto de mini apps independientes que comparten infraestructura.

Convenciones existentes:

- Las rutas viven en src/app.
- La lógica de dominio vive en src/features/<feature-name>.
- La DB usa Prisma en prisma/schema.prisma.
- El cliente Prisma compartido está en src/lib/db.ts.
- Las mutaciones deben implementarse como server actions.
- Las server actions deben validar inputs con zod.
- Las rutas deben mantenerse delgadas e importar data/components desde src/features.
- El diseño actual usa app-shell, soft-grid, panel, panel-muted, glass-button y liquid-text.

La nueva mini app debe seguir esta estructura:

- src/app/portfolio/page.tsx
- src/features/portfolio/data.ts
- src/features/portfolio/actions.ts
- src/features/portfolio/types.ts
- src/features/portfolio/services.ts
- src/features/portfolio/components/

También actualizar src/app/page.tsx para agregar Portfolio al dashboard principal.

Objetivo del producto
=====================

Portfolio debe trackear inversiones, activos manuales, pasivos y net worth histórico.

La app debe permitir:

- Registrar activos de mercado, como crypto, acciones, ETFs, fondos y commodities/metales.
- Registrar transacciones reales de compra, venta, transferencias y ajustes.
- Calcular holdings actuales a partir de transacciones.
- Actualizar precios de mercado automáticamente al cargar la página si están vencidos.
- Registrar activos manuales como cash, propiedades, vehículos u otros bienes.
- Registrar pasivos/deudas manuales.
- Convertir todos los valores a una moneda de display seleccionable: USD o UYU.
- Usar USD como moneda de display por defecto.
- Obtener tipo de cambio USD/UYU automáticamente por API.
- Calcular total assets, total liabilities y net worth.
- Guardar snapshots históricos diarios para graficar la evolución.
- Permitir checkpoints manuales con nota opcional.
- Mostrar performance: costo promedio, P/L, allocation por categoría y cambios por período.

Decisión clave de arquitectura
==============================

Los activos de mercado deben ser transaction-first.

La cantidad actual de un activo como BTC, AAPL o VOO no debe ser la fuente de verdad editable principal. La fuente de verdad son sus transacciones.

Ejemplo:

- Si registro una compra de BTC, se suma cantidad.
- Si registro una venta de BTC, se resta cantidad.
- Si registro una transferencia o ajuste, se modifica cantidad según corresponda.

Se puede guardar quantityHeld o currentTotalValue como cache en PortfolioAsset, pero deben recalcularse después de crear, editar o archivar transacciones.

Modelo de costo y P/L
=====================

Usar weighted average cost para el MVP.

No implementar FIFO en el MVP.

La app debe calcular:

- quantityHeld
- averageCost
- costBasis
- marketValue
- unrealizedGain
- unrealizedGainPercent

Reglas:

- averageCost = costBasis / quantityHeld
- marketValue = quantityHeld * currentUnitPrice
- unrealizedGain = marketValue - costBasis
- unrealizedGainPercent = unrealizedGain / costBasis

El diseño de datos no debe impedir implementar FIFO en el futuro, pero no hay que construir FIFO ahora.

Tipos de activos
================

Internamente, distinguir:

- MARKET_ASSET
- MANUAL_ASSET
- LIABILITY

Visualmente, no mostrar "manual assets" como categoría principal. Mostrar categorías útiles:

- Crypto
- Stocks
- ETFs & Funds
- Commodities
- Cash
- Real Estate
- Vehicles
- Other Assets
- Liabilities

Cash debe ser un activo manual con categoría visual Cash.

Detalles como "emergency fund", "checking account", "cash wallet", "banco X" o "cuenta Y" deben ir en notes o accountNote, no como categoría separada.

Símbolos
========

Los símbolos deben guardarse uppercase y trimmed.

Crypto debe registrarse con símbolos simples:

- BTC
- ETH
- SOL

No usar pares como BTC-USD como input principal.

Acciones, ETFs y fondos deben soportar símbolos USA/internacionales:

- AAPL
- MSFT
- VOO
- SPY

Commodities deben prepararse para precio spot global:

- XAU para oro
- XAG para plata

No limitar commodities a ETFs como GLD o SLV, aunque esos ETFs pueden funcionar como market assets normales si se registran como ETF.

Monedas
=======

Soportar:

- USD
- UYU

USD debe ser la moneda de display por defecto.

El usuario debe poder registrar activos, transacciones y pasivos en USD o UYU.

La pantalla debe convertir todos los valores a la moneda de display seleccionada.

El tipo de cambio USD/UYU debe obtenerse automáticamente por API y cachearse en DB.

La actualización del FX debe seguir una lógica similar a los precios de mercado:

- Si no hay tipo de cambio guardado, buscarlo.
- Si el último tipo de cambio está stale, buscarlo.
- Si falla la API, usar el último tipo de cambio guardado y mostrar warning no bloqueante.

Actualización automática al cargar /portfolio
=============================================

Al entrar a /portfolio, ejecutar una función tipo getPortfolioPageData().

Esta función debe correr server-side, no desde el client directamente.

Flujo esperado:

1. Asegurar que existe PortfolioSettings.
2. Leer activos, transacciones, pasivos, últimos precios, último FX y últimos snapshots.
3. Detectar activos de mercado con autoPriceEnabled = true.
4. Considerar precio stale si lastPriceUpdatedAt es null o si pasaron más de priceRefreshHours.
5. Actualizar precios stale antes de devolver la data de pantalla.
6. Actualizar tipo de cambio USD/UYU si está stale.
7. Recalcular holdings desde transacciones.
8. Convertir todos los valores a displayCurrency.
9. Crear snapshot diario si el último snapshot automático tiene más de snapshotIntervalHours.
10. Devolver page data ya lista para renderizar.

No romper la pantalla si falla una API externa.

Si una API falla:

- Usar último precio guardado.
- Marcar el asset o FX como stale/error.
- Mostrar warning compacto.
- Renderizar el portfolio igual.

Transacciones
=============

La UI debe tener una acción principal para activos de mercado: Add transaction.

Tipos mínimos:

- BUY
- SELL
- TRANSFER_IN
- TRANSFER_OUT
- ADJUSTMENT

BUY
---

Debe soportar dos modos de input:

Modo by total amount:

- User ingresa grossAmount, unitPrice, fees.
- El sistema calcula quantity.

Formula:

quantity = grossAmount / unitPrice

Fees aumentan cost basis.

Modo by asset quantity:

- User ingresa quantity, unitPrice, fees.
- El sistema calcula grossAmount.

Formula:

grossAmount = quantity * unitPrice

El unitPrice debe autocompletarse con el último precio de mercado si existe, pero siempre debe ser editable.

SELL
----

Debe soportar dos modos de input:

Modo by asset quantity:

- User ingresa quantity, unitPrice, fees.
- El sistema calcula grossAmount.

Modo by total amount:

- User ingresa grossAmount, unitPrice, fees.
- El sistema calcula quantity.

Fees reducen net proceeds.

Bloquear ventas si quantity vendida excede quantityHeld actual.

TRANSFER_IN / TRANSFER_OUT
--------------------------

Se usan para reflejar movimientos externos o correcciones de custodia.

Deben afectar quantityHeld.

No deben tratarse como profit/loss por defecto.

ADJUSTMENT
----------

Se usa para correcciones manuales.

Debe requerir notes.

Debe afectar quantityHeld.

Reglas de holdings
==================

Para cada asset de mercado:

quantityHeld =
  sum(BUY.quantity)
  + sum(TRANSFER_IN.quantity)
  - sum(SELL.quantity)
  - sum(TRANSFER_OUT.quantity)
  + sum(ADJUSTMENT.quantity)

Si quantityHeld queda 0, el asset puede seguir visible hasta que el usuario lo archive manualmente.

No archivar automáticamente assets en 0.

Pasivos
=======

Para el MVP, pasivos son simples.

Campos:

- name
- type
- currency
- currentBalance
- accountNote
- notes

No implementar cuotas, vencimientos, calendario de pagos ni tasa de interés en MVP.

El objetivo principal de pasivos es calcular net worth.

Si currentBalance llega a 0, no archivar automáticamente. El usuario debe archivarlo manualmente.

Archivo y borrado
=================

No hacer hard delete desde la UI.

Assets, transactions y liabilities deben poder archivarse con archivedAt.

Los elementos archivados no deben aparecer en la vista principal.

Los elementos archivados deben mantenerse en DB para preservar historial.

Snapshots históricos
====================

Crear snapshots automáticos diarios.

Regla:

- Al cargar /portfolio, si el último PortfolioSnapshot automático tiene más de snapshotIntervalHours, crear uno nuevo.
- Por defecto snapshotIntervalHours = 24.

También permitir snapshot manual:

- Acción secundaria o poco prominente.
- Label sugerido: Create checkpoint.
- Debe permitir note opcional.
- source = MANUAL.

Los snapshots deben guardar:

- totalAssets
- totalLiabilities
- netWorth
- currency
- source
- note opcional
- capturedAt

Usar snapshots para mostrar cambio histórico.
